# Extraction — two prompts: chats and commits

Knowledge enters from two streams with genuinely different jobs, so there are **two
separate prompts**. The chat prompt captures durable non-derivable knowledge that
surfaced in discussion — decisions, constraints, gotchas, glossary terms, product
context — **whether or not any code changed** (a session can settle a decision with zero
diff; that's a first-class keeper). The commit prompt does *code-synthesis* (refresh
derivable caches) and flags drift, from the diff alone. Run them independently — they
have different inputs, cadences, and auto-apply safety.

## Run extraction off the main thread

Transcripts and diffs can be long, and the triage is token-heavy. **Dispatch each
extraction to a subagent** (give it the `kb.py outline`, the transcript/diff, and the
relevant prompt below); it returns proposed page diffs without bloating the main
session's context. For `kb catchup` over several sessions, fan them out — each session's
triage is independent, so run them as **parallel subagents** or a **background job** and
collect the proposals when done. Don't block the user while a 2,000-line transcript is
mined.

## Always first: load the wiki structure into context

Both prompts route and dedup against the **actual** wiki — its real categories
(including any repo-specific deviations recorded in the root `INDEX.md`) and its existing
pages — not the generic defaults. So before either prompt, load:

```bash
python3 scripts/kb.py outline
```

This emits the root `INDEX.md` (resolver + manual + deviations), each folder's purpose,
and every existing page with its Compiled-Truth first line. **Route with these
categories; dedup against these pages.** Without this, the agent invents structure and
duplicates pages.

## Prompt 1 — Chat extraction (durable knowledge from discussion)

Captures non-derivable knowledge that surfaced in the conversation — **decisions,
constraints, gotchas, glossary terms, product/user context, mental models** — **whether
or not any code changed this session.** A pure design discussion, a brainstorm that
settles direction, or a decision reached before a line is written all produce keepers.

**Inputs:** WIKI OUTLINE (above) · TRANSCRIPT (`scripts/vendor/recall/read_session.py
<file>`) · DIFF (**optional** — `git diff <watermark-sha>..HEAD`; **may be empty**; used
only as a hint for focus and for guessing `covers`).

> You are extracting durable, non-derivable knowledge from a work session into a repo's
> knowledge base. You have the wiki's current structure and pages, the chat transcript,
> and optionally a git diff (which **may be empty — many valuable sessions change no
> code**).
>
> **Keep only knowledge that is all three of:**
> - **Non-derivable** — cannot be regenerated by reading the code: a decision + its
>   rejected alternatives + rationale, a constraint, a gotcha, load-bearing weirdness, a
>   mental model, product/user context, the meaning of a domain term.
> - **Durable** — still matters next month. Not transient, not status.
> - **Relevant to the repo** — it shapes how this project is understood or evolves. It
>   need **not** correspond to a code change; **a decision reached with zero diff is a
>   first-class keeper.**
>
> **Discard:** restatements of what the code plainly says; anything rederivable by
> reading the repo; chit-chat; tool output; anything already in the wiki outline. **Be
> stingy — most sessions yield 0–3 items. If nothing durable was settled, propose
> nothing.**
>
> **For each kept item:**
> 1. Route it using the **categories in the wiki outline** (honor deviations).
> 2. `source: canonical` (knowledge born in the discussion).
> 3. Set `covers` to the **code area the knowledge is *about*** — NOT "what changed." A
>    constraint about billing → `src/billing/**` even if billing wasn't touched this
>    session (so drift fires when it eventually changes). Leave `covers: []` when the
>    knowledge isn't about specific code — a process/product decision, or a decision that
>    predates the code. The diff, if any, is only a hint for guessing this.
> 4. Set `verified_against` to the current HEAD sha. Terse, claim-first **Compiled
>    Truth** + one **Timeline** entry:
>    `- <date> — captured from chat (session <id>) — @<HEAD-sha>`.
> 5. **Dedup against the outline:** if a page already covers this, propose an *update*
>    (append Timeline, rewrite Compiled Truth), not a new page. For a decision, record
>    the rejected alternative(s) + rationale; supersede prior decisions by reference,
>    never edit their history.
>
> **Output:** proposed page diffs, each with a one-line summary, for human approval. Do
> not write any file.

**When:** at session-end (warm fast-path) and inside `kb catchup` (the safety net) for
each session since the watermark — **including sessions with no commits**, which the
chat watermark enumerates independently of git.

## Prompt 2 — Commit extraction (code-synthesis + drift)

**Inputs:** WIKI OUTLINE (above) · DIFF (`git diff <base-sha>..HEAD`). **No transcript** —
this prompt reads only the code change, so it can run at commit/PR/CI time with no chat.

> You are reconciling a repo's knowledge base with a code change. You have the wiki's
> current structure and pages, and the git diff. The diff is the source of truth for what
> the code now *is*.
>
> Do two things:
>
> **A. Refresh derivable caches.** For each `from-code` page in the outline whose `covers`
> intersect the diff, propose a re-synthesis: re-trace the now-current code and rewrite
> its Compiled Truth, appending a Timeline entry `- <date> — re-synthesized from code —
> @<sha>`. If the diff introduced a subsystem whose cross-file flow is **expensive to
> re-trace and will be read often**, propose a new `from-code` page in `architecture/`.
> Do **not** create pages for cheap-to-read code — derivable knowledge is generated on
> demand, not persisted.
>
> **B. Flag canonical drift.** For each `canonical` page whose `covers` intersect the
> diff, do **not** rewrite it — its truth isn't in the code. Instead flag it for human
> re-verification with a specific question, e.g. "this constraint covers
> `src/billing/**`, which changed — is '<Compiled Truth>' still true?"
>
> **Output:** proposed `from-code` page diffs (A) and a review list (B), for human
> approval. Never auto-apply a canonical rewrite.

**When:** post-commit hook, pre-PR, or CI. This is the git stream's own contribution —
independent of any chat. (`kb status` computes the same `covers`∩diff intersection
deterministically; this prompt adds the synthesis + the specific re-verification
question.)

## Why the split

| | Prompt 1 (chat) | Prompt 2 (commits) |
|---|---|---|
| Input | transcript (diff optional, may be empty) | diff only |
| Captures | non-derivable knowledge — decisions, constraints, terms, gotchas | refreshes derivable caches; flags drift |
| Needs a code change? | **no** — zero-diff discussions count | yes — it *is* the diff |
| Needs a chat? | yes | **no** — runs at commit/PR/CI |
| Auto-apply | propose-only (canonical) | A: re-synth is safe to draft; B: flag-only |
| Precious? | capture-or-lose-forever | regenerable cache |

## Dedup helpers (both prompts)

```bash
python3 scripts/vendor/recall/recall.py "<topic>" --project "$(pwd)"   # past sessions
grep -ri "<topic>" repo-wiki/                                          # existing pages
```
